为什么 Writing Agent 看不到全部 Canon？

长篇小说里"什么人知道什么"是核心戏剧张力。如果 Writing Agent 把 Canon 全部塞进 prompt，模型会自然把"作者视角的全部真相"写进角色对话或心声，造成穿帮。所以 Context Compiler 先按本章场景挑出可见子集，把"角色 A 还不知道 X"这件事钉死，Writing Agent 写的时候就没机会泄露。

Bootstrap：导入前三章 → 自动生成初始 Canon

一本书的起点。用户给 3 段原文，系统按模板填出 4 类基础文件。这一步只跑一次。
输入

用户从已有手稿或开头试写中粘 3 段 markdown，每段独立保存到 imports/。

POST /bootstrap/import

Import Agent

调 LLM 按"角色卡 6 个模板槽 / WB 主题分类 / KM 5 种状态 / Memory 时间线"格式抽信息。强 Pydantic schema 校验。

手动补齐 / 修正

面板里直接改 6 个角色卡模板槽 / KM 可见性 / WB 条目。改完才算正式起点。

每次改写 canon_edit_history
为什么需要"模板"？

v1.0 之前 LLM 自己总结的 Character / KM / Memory 太散：长短不一、字段名乱跳、有时候把"状态"写到"性格"里。改造后：角色卡固定 6 个槽（基本信息 / 稳定人格 / 当前状态 / 说话方式 / 关系 / 知道什么），LLM 必须按槽填，schema 不对就重试，重试两次都不行才报错给用户。看起来"约束多"，其实是让 LLM 干"它擅长的填空"，不让它干"它不擅长的开放写作"。

4

章节生成详细链路：从一句方向到 Canon 落库

这是真正每章都要跑一遍的链路。颜色 = 这一段谁在干活。
用户 · Step 1填入本章方向
prompt: string→ POST /chapters/{id}/prompt
没有任何 Canon 写入，只暂存到 chapter.draft_prompt。
Agent · 后台 1整理用户意图
Intent Parser→ 显式实体 / 情绪基调 / 是否允许新人物
Context Compiler→ 从 Canon 抽子集，构造 allowed_named_entities
Prompt Expander→ 输出结构化 Prompt（含本章硬约束）
用户 · Step 2审 Prompt
看结构化 Prompt + 安全边界摘要→ 点"确认"
允许返回 Step 1 改方向；批准则进入 Writing。
Agent · 后台 2SSE 流式写正文
Writing Agent→ SSE 边写边推到前端
前端可中断；中断不消耗 token 之后的部分。
Audit · 并发 3审正文
Named Entity 正文里的专名是否都在白名单
Knowledge 角色是否泄露了他不该知道的事
Continuity 是否改写已落地事实 / 时间线穿越
三个并行跑，结果合并成 audit_report，含 highest_severity。
Agent · 后台 3S0 触发自动 Revision
audit 有 S0 ?→ 是：Revision Agent 自动重写（最多 2 次）
2 次后仍有 S0 ?→ 是：阻拦批准，提示用户人工介入
用户也可以主动写 feedback 触发 Revision，与 S0 自动 Revision 共用一条链路。
用户 · Step 3 / 4审正文 → 批准
看正文 + audit_report→ 提反馈 / 直接批准
仅当 S0 = 0 时，"批准正文"按钮才可点。
Agent · 后台 4从正文提取候选事实
Extraction Agent→ 按四个目标分类候选 patch
Canon Merge Agent→ 与现 Canon 比对，标 conflict / new / update
用户 · Step 5确认 Canon 更新
看 patch 列表，按 target badge 分别确认→ 提交
Canon · 落库写四张基础表
World Bible / Characters / KM / Memory 四张表同事务写入
每条都新增一行 canon_edit_history，source = canon_merge_agent
本章 status → finalized，current_canon_version +1
5

出错怎么救：每种失败都要有"下一步"

v1.0 之前所有错误都崩成 502 空白。改造后每种错对应一个明确动作。
触发场景
系统行为
用户看到 / 能做
LLM 超时 / 5xx
LLM Gateway 自动重试 2 次（指数退避）
不打扰用户；2 次后报 "上游不稳定"，按钮变"重试"
LLM 429 限流
退避 + 切备用模型（如配置了）
提示 "切换到备用模型 X 继续"
LLM 401 / API key 失效
不重试，直接报
红色横幅：去 Settings 校 key，给跳转按钮
JSON 解析失败 / schema 不符
请求 LLM 重出（带"上次错在哪"提示），最多 2 次
2 次仍失败：展原始输出 + "保留为草稿"
Audit S0 violation
自动 Revision 2 次
仍未清：阻拦"批准正文"，列具体违规位置
Canon merge conflict
不自动落库，挂 pending_user_confirmation
逐条 patch 让用户选 "接受 / 跳过 / 改"
本地后端宕
App 健康检查 5s 一次
顶部黄条：本地服务未运行，给"启动服务"按钮
本地 DB 写失败
事务回滚 + 错误入 agent_runs.error_message
提示 "本章未保存，建议立即备份" + 跳备份脚本

7

一句话总结

如果只能记住一句话，就记这句。
v1.0 的本质，是把"一条全是 mock 的样品产线"换成"一条用户只做 5 件事、9 个 Agent 各司其职、Writing Agent 看不到全部 Canon、每步都能失败也能救回来的真正产线"——并且只在用户本机跑。
